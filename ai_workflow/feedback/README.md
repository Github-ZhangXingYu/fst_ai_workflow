# 测试反馈目录 — 维护说明

> 本目录记录 FST AI 测试工作流在 CentOS 9 环境下的运行反馈。
> 每次工作流执行完毕后，用户与 Claude Code 交流，由 Claude Code 按下方模板写入新反馈。
> 用户将反馈传回 Windows 开发侧后，Claude Code 读取最新反馈来定位和修复工作流问题。

## 目录结构

```
ai_workflow/feedback/
├── README.md                    # 本文件
├── FST-WF-TEST-NNN.md           # 最新一次测试反馈（只有一份，不堆叠）
└── archive/                     # 所有历史反馈归档
    └── FST-WF-TEST-NNN.md
```

## 命名规范

所有文件使用 `FST-WF-TEST-{NNN}.md`，NNN 为三位递增序号（001, 002, …）。

## 添加新反馈流程

1. 将 `ai_workflow/feedback/` 根目录下当前的 `FST-WF-TEST-*.md` 移入 `archive/`
2. 查看 `archive/` 中最大序号，+1 作为新序号
3. 按下方模板写入新反馈
4. 更新本 README 末尾的索引表

---

## 反馈报告模板

新反馈请严格按此模板编写。

> **定位**：本模板由 CentOS 侧的 Claude Code 填写。你只需记录**现象**和**可能原因分析**，不需要给出确定的修复方案——修复决策由用户拿到 Windows 开发侧后，与更强的模型逐个讨论决定。

```markdown
# FST-WF-TEST-NNN

| 项目 | 值 |
|------|-----|
| **日期** | YYYY-MM-DD |
| **工作流 ID** | xxx |
| **目标模块** | xxx |
| **分支** | xxx |
| **结论** | ✅ 无问题 / ⚠️ 可跑通但有问题 / ❌ 阻断 |
| **活跃耗时** | ~X 小时 |
| **阻塞耗时** | ~X 小时（如有） |
| **Token 消耗** | orchestrator: xxx, 子 Agent: xxx（估算） |
| **环境** | CentOS Stream 9 |

## 问题清单

> 按 P0 → P1 → P2 → P3 排序。每个问题只需写清现象和可能原因，不必给出最终修复方案。

### P0-1: [问题标题]

| 项 | 内容 |
|----|------|
| **阶段** | 步骤 X |
| **现象** | 实际发生了什么，有错误信息的话附上 |
| **可能原因** | 你的分析/猜测，不必确信，列出你认为最可能的 1~2 个原因 |
| **状态** | ❌ 待修复 / ✅ 本次已修复 |

### P1-1 / P2-1 / P3-1: ...

同上格式。
```
